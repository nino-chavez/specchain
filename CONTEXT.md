# specchain — Domain Glossary

This is the canonical vocabulary for specchain. Use these terms exactly. Operational rules live in [`README.md`](./README.md) and the per-command instructions in [`.claude/commands/specchain/`](./.claude/commands/specchain/).

---

## The pipeline

specchain is a three-phase, spec-driven workflow. The phases are explicit, named, and not interchangeable:

| Term | Definition |
|---|---|
| **`/new-spec`** | Phase 1. Initialize a dated spec folder, run requirements Q&A, persist the execution profile. Output: `planning/requirements.md`, `planning/execution-profile.yml`. |
| **`/create-spec`** | Phase 2. Generate `spec.md` (spec-writer) and `tasks.md` (tasks-list-creator), optionally verify (depends on depth). Output: `spec.md`, `tasks.md`. |
| **`/implement-spec`** | Phase 3. Execute task groups via solo or squad strategy, run domain verifiers, update `STATE.md`. Output: `implementation/*`, `verification/*`. |

## Execution profile axes

Every spec has a profile with two orthogonal axes:

| Term | Definition |
|---|---|
| **Strategy** | `solo` (single agent handles all task groups, vertical slices) or `squad` (multi-agent delegation, domain-layer groups). |
| **Depth** | `lean` (3-5 questions, skip verifiers, minimal tests), `standard` (default), or `thorough` (max rigor — extra arch/test questions, TDD red-green-refactor template per group). |

Strategy and depth are independent: `solo + lean`, `squad + thorough`, etc. are all valid combinations.

## Task taxonomy

| Term | Definition |
|---|---|
| **Task group** | A unit of work assigned to one implementer. Squad mode groups by domain layer (Database, API, Frontend, Testing); solo mode groups by feature slice. |
| **Implementer** | A subagent role declared in `specchain/roles/implementers.yml` with declared `areas_of_responsibility`. The tasks-list-creator matches groups to implementers. |
| **AFK** | A task group an implementer can complete unattended and merge. *(Adopted from matt-pocock/skills `to-issues`.)* |
| **HITL** | A task group requiring human-in-the-loop interaction (judgment call, design review, credential, approval gate). Must include a `HITL reason` line. |
| **Proof of Life** | The single smallest end-to-end user action that proves the feature works. The first implementation group MUST deliver this — not types, not infrastructure, not a model alone. |

## Spec folder structure

```
specchain/specs/[YYYY-MM-DD-feature-name]/
├── planning/
│   ├── initialization.md         # Raw user idea
│   ├── requirements.md           # Q&A results
│   ├── execution-profile.yml     # Strategy + depth
│   └── visuals/                  # Mockups, screenshots
├── spec.md                       # Generated by spec-writer
├── tasks.md                      # Generated by tasks-list-creator
├── implementation/               # Per-implementer reports
└── verification/                 # Domain verifier reports
```

## State and memory

| Term | Definition |
|---|---|
| **STATE.md** | The session memory file at `specchain/STATE.md`. Tracks blockers, decisions, patterns, execution log across multiple `/implement-spec` runs. |
| **Verification report** | Output from a domain verifier (backend, frontend, etc.) confirming a task group's outputs meet spec. Stored in `verification/`. |

## Avoid (vocabulary drift)

These terms have caused confusion — replace if encountered:

- "ticket" / "issue" → use **task group** (specchain doesn't write to issue trackers)
- "subtask" alone is ambiguous → say **task** for a numbered item, **task group** for the parent unit
- "rigor level" → use **depth** (the named axis)
- "mode" → use **strategy** (solo/squad) or **depth** (lean/standard/thorough), depending on which axis you mean
- "spec" alone is ambiguous → say **spec folder** (the directory), **spec doc** or **`spec.md`** (the file inside it), or **spec name** (the dated identifier)
